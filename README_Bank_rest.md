# Bank Cards Management System (MVP)

Backend-приложение для управления банковскими картами, счетами и пользователями.  
Реализована аутентификация через JWT, ролевая модель (ADMIN/USER), шифрование номеров карт (AES‑GCM), переводы между своими картами, поиск и пагинация.

## Основные возможности
- Регистрация и аутентификация пользователей (JWT).
- Администратор:
    - создание, блокировка, активация, удаление карт;
    - управление пользователями (CRUD, блокировка/активация);
    - просмотр всех карт с пагинацией.
- Пользователь:
    - просмотр своих карт (поиск по последним 4 цифрам + пагинация);
    - запрос на блокировку карты;
    - переводы между своими картами;
    - просмотр баланса карты.
- Шифрование номера карты (AES‑GCM) и маскировка в ответах (`**** **** **** 1234`).
- Поиск карт по хешу последних 4 цифр (без расшифровки).
- Валидация входящих данных и глобальная обработка ошибок.
- Документация API через Swagger/OpenAPI.

## Настройка базы данных
1. Создайте базу данных, например `bank`:
   ```sql
   CREATE DATABASE bank;
   ```
2. Убедитесь, что параметры подключения в `src/main/resources/application.properties` соответствуют вашей БД. Пример:
   ```properties
   spring.datasource.url=jdbc:postgresql://localhost:5432/bank
   spring.datasource.username=postgres
   spring.datasource.password=вставьте_свой_пароль
   spring.jpa.hibernate.ddl-auto=create
   ```
    - При первом запуске Hibernate создаст таблицы автоматически (`ddl-auto=create`). После первого успешного запуска рекомендуют переключиться на `validate` или использовать Liquibase.
    - Liquibase подготовлен (скрипты в `src/main/resources/db/changelog`). Чтобы включить Liquibase:
        - установите `spring.liquibase.enabled=true`
        - установите `spring.jpa.hibernate.ddl-auto=none`

3. Шифрование номера карты требует 256‑битного ключа AES в Base64: параметр `app.crypto.key` в `application.properties`. Сгенерируйте и подставьте ключ, если он отсутствует.

## Аутентификация и роли
- При первом запуске автоматически создаётся администратор:
    - Email: `admin@example.com`
    - Пароль: `admin123`
- Получение JWT‑токена — пример запроса:
  ```
  POST /api/v1/auth/login
  Content-Type: application/json

  {
    "email": "admin@example.com",
    "password": "admin123"
  }
  ```
- Ответ содержит токен:
  ```json
  {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
  ```
- Для запросов, требующих авторизации, добавляйте заголовок:
  ```
  Authorization: Bearer <ваш_токен>
  ```

## Безопасность
- Пароли пользователей хранятся в виде BCrypt‑хеша.
- Номера карт шифруются алгоритмом AES‑GCM (256 бит) с уникальным IV для каждой записи.
- Поиск карт по последним 4 цифрам осуществляется через HMAC‑хеш, что позволяет искать без расшифровки номеров.
- JWT‑токены подписываются секретным ключом, указанным в `application.properties`.

## Возможные проблемы и решения
- Ошибка подключения к БД: проверьте, что PostgreSQL запущен и параметры в `application.properties` верны.
- Шифрование номера карты: убедитесь, что `app.crypto.key` задан (256‑битный ключ в Base64). Если нет — сгенерируйте новый и подставьте.
- JWT не валиден: проверьте время жизни токена и корректность заголовка `Authorization`.
